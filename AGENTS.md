## 项目定位

本项目是 `cotton-filter`，目标是帮助棉花贸易客户从 Excel 棉花资源表中智能筛选出符合规则的棉花批次。

客户业务背景：

- 客户主要通过 Excel 查看棉花资源，筛选合适批次后买入，再卖给纺织厂、贸易公司等下游客户赚差价。
- 棉花贸易属于高流水、低净利的大宗商品贸易。流水可能很大，但单吨利润很薄，所以系统价值在于更快、更准、更低风险。
- 当前项目优先做“智能筛选/批次匹配”，不是先做完整 ERP、交易系统或复杂 AI。

## 核心业务流程

典型流程：

```text
收到客户询价
-> 查看 Excel/资源表
-> 按长度、强力、马值、基差、仓库、加工类型等条件筛选
-> 计算或参考升贴水/得分
-> 找出符合标准且有利润空间的批次
-> 报价
-> 点价/签合同
-> 提货/结算
```

系统第一阶段应服务于：

- 快速筛选符合客户要求的批次。
- 减少人工翻 Excel 和人工判断。
- 保留 Excel 导入导出能力，贴合客户现有工作方式。

## 棉花行业关键概念

- `批号`：一批棉花的唯一标识，后续出库、结算、开票都依赖它。
- `仓单/资源`：可销售棉花资源的数字凭证或资源记录。
- `点价合约`：如 `CF2609`，表示郑棉 2026 年 9 月合约。
- `基差`：现货价与期货价的差值。
- `现货价`：通常可理解为 `期货价 + 基差`。
- `升水/贴水`：棉花品质相对基准品更好则加价，更差则减价。
- `点价`：先约定基差，后续再按某个期货价格结算。
- `套保`：通过期货对冲现货价格波动风险。

## 资源表常见字段

Excel 表里常见字段包括：

- 价格/交易：`挂单价`、`点价合约`、`基差`
- 标识：`批号`
- 来源：`加工单位`、`加工类型`
- 重量：`包数`、`公重`、`毛重`
- 品质：`颜色级`、`长度`、`比强/强力/断裂比强度`、`马值/马克隆值`、`含杂`、`回潮`、`长整/整齐度`
- 仓储：`仓储地`、`仓库`、`库费承担`
- 状态：是否已售、是否点价、是否出库、是否结算

新增模板支持时，优先通过前端“规则维护”菜单或本地 SQLite 规则库维护列名规则，不要再把某一家公司的列名写死到 Python 常量里。

## 升贴水规则背景

基准品：

- 颜色级：`31`
- 长度：`28mm`
- 马克隆值：`B1/B2`
- 断裂比强度：`S3`
- 长度整齐度：`U3`
- 扎工质量：`P2`
- 异性纤维：未超过 1 包
- 基准升贴水：`0 元/吨`

常见升贴水参考：

- 颜色级：`11 +350`，`21 +200`，`31 0`，`41 -350`，`12 -300`，`22 -500`
- 长度：`>=30mm +400`，`29mm +150`，`28mm 0`，`27mm -300`
- 马克隆值：`A +100`，`B1/B2 0`，`C2 -100`
- 断裂比强度：`S1 +250`，`S2 +200`，`S3 0`，`S4 -250`
- 长度整齐度：`U1 +250`，`U2 +200`，`U3 0`，`U4 -200`
- 扎工质量：`P1 +100`，`P2 0`，`P3 -300`
- 异性纤维：超过 1 包后，每多 1 包贴水 `200 元/吨`

当前代码里的动态数据规则是一套简化评分逻辑，不等同于完整交易所/收购规则。若要改规则，优先让用户确认客户实际 Excel 规则和验收样例，再通过规则库维护。

## 当前实现约束

主要文件：

- `backend/server.py`：Python FastAPI 本地后端入口，作为 Tauri sidecar 启动。
- `cotton_filter_app/constants.py`：共享应用常量。
- `cotton_filter_app/rules.py`：本地 SQLite 规则库、默认规则初始化、列名规则和数据规则 CRUD。
- `cotton_filter_app/text_utils.py`：表头/规则文本标准化。
- `cotton_filter_app/header.py`：Excel 表头识别和字段映射。
- `cotton_filter_app/scoring.py`：棉花资源评分规则。
- `cotton_filter_app/processor.py`：sheet/workbook 处理和输出表生成。
- `cotton_filter_app/file_utils.py`：批量文件、输出路径和文件展开工具。
- `src/App.tsx`：Tauri Web 前端主界面和顶部菜单。
- `src/RulesView.tsx`：列名规则、数据规则维护页面。
- `src/backend.ts`：通过 Tauri shell plugin 启动/停止 Python sidecar。
- `src/api.ts`：前端调用本地 FastAPI 的接口封装。
- `src-tauri/tauri.conf.json`：Tauri v2 应用、sidecar、bundle 和 updater 配置。
- `src-tauri/capabilities/default.json`：Tauri dialog/opener/shell/updater 权限。
- `scripts/build-backend.mjs`：用 PyInstaller 构建 Python sidecar，并按 Tauri target triple 放入 `src-tauri/binaries`。
- `scripts/write-updater-manifest.mjs`：根据 Tauri updater 产物生成 GitHub Release 使用的 `latest-{{target}}-{{arch}}.json`。
- `.github/workflows/build.yml`：GitHub Actions 构建流程，仅在推送 `v*` 版本 tag 或手动触发时运行；安装 Node、Rust、Python 3.10，先运行 Python 测试，再构建 Python sidecar、Vite 前端和 Tauri 桌面应用；若配置了 `TAURI_SIGNING_PRIVATE_KEY`，会额外生成 updater 签名产物和 manifest。

当前处理逻辑：

- 支持 `.xlsx` 和 `.xls`。
- 自动扫描前置抬头，定位真实表头行。
- 通过本地 SQLite 的列名规则对齐多家公司模板；表头标准化后精确匹配，匹配成功即映射为统一字段。
- 支持两行组合表头；例如 `颜色级比例(%)` 下分 `白棉2级`、`白棉3级` 等多列时，会合并成统一颜色级占比文本再评分。
- 颜色级占比支持 `95%`、`95％`、`31:84.4` 等写法；单独写 `白棉3级` 或 `31` 时按数据规则里的颜色级值别名处理为 `100%`。
- 颜色级值别名必须区分“匹配值”和“输出值”：匹配值是 Excel 中多变的别名，如 `白棉3级`、`白棉三级`、`31`、`三级棉`；输出值是统一标准值，如这些别名都应输出为 `白棉3级`。默认规则和已有本地规则库初始化时都会按此逻辑修复颜色级标准输出值。
- 数值字段支持从 `30.5mm`、`4.1(A)`、`打包-1500` 等文本中提取数字。
- 必需字段是 `基差`、`长度`、`马值`。
- 对每行按 SQLite 数据规则计算 `得分`，再计算 `与基差差距 = 基差 - 得分`。
- 默认初始化不再内置旧的评分区间或过滤区间；后续由用户在规则维护里新增评分区间、过滤区间或关键词过滤。
- 用户维护的列名规则和数据规则持久化在本机用户数据目录的 `rules.sqlite3` 中；正常自动更新或覆盖安装新版应用不会删除该规则库。只有更换 `APP_NAME`、手动指定新的 `COTTON_FILTER_RULE_DB`、删除用户数据目录或清空规则库时，已有规则才会丢失或迁移不到。
- 导出结果工作簿包含两个 sheet：`筛选结果` 使用原始 Excel 列名和原始单元格数据展示命中行，并追加 `得分`、`与基差差距`；`识别异常` 展示列名规则未覆盖、数据值别名未覆盖、基差无法解析等问题，并尽量附带原始行数据。
- 输出到 `cotton-filter-results`，文件名使用原文件名；重名时追加 `_2`、`_3` 等编号。
- GUI 处理日志保持精简：只展示开始筛选、每个文件成功/失败、筛出行数和最终汇总。
- GUI 采用 Tauri + React + TypeScript 实现，Tauri 负责窗口、安装包、自动更新、文件选择、打开目录、启动/停止后端；Python 只负责 Excel 解析、字段匹配、评分、筛选、写出结果。
- GUI 顶部菜单包含“文件筛选”和“规则维护”；规则维护页只保留“列名规则”和“数据规则”两个主体区域。
- 列名规则页面按标准字段分组展示，标准字段不可编辑；用户只能给某个标准字段新增别名或删除已有别名，新增别名必须通过弹框录入。
- 列名规则不在页面暴露启用/停用状态。
- 数据规则页面不再暴露“规则名称”；用户只维护规则类型、匹配值、区间、加减分、输出值等实际规则条件，后端会为兼容旧数据自动生成内部规则名称。
- 规则维护页采用撑满页面的左右布局：左侧菜单包含“列名规则”“数据规则 / 值别名”“数据规则 / 评分与过滤区间”，右侧展示当前菜单对应的数据，菜单和内容区域各自内部滚动。数据规则中的“值别名”要与“评分与过滤区间”分开维护；值别名按输出值归组并单列展示；评分区间和过滤区间必须选择同一字段值别名列表中的“输出值”，不能留空或自由填写；最小值和最大值支持纯数值与百分比写法，如 `80`、`80%`、`80％` 都按数值 `80` 保存；关键词过滤命中指定字段文本后保留该行。
- 规则维护页右上角提供一键导出/导入全部规则；导出格式为 JSON，不包含 SQLite 自增 ID，导入时会整体替换当前列名规则和数据规则，并重新计算匹配 key。
- 前端保持克制的桌面工具样式：浅色工作台、表格式文件列表、浅色日志区和少量墨绿色状态强调；不要恢复 Tkinter。
- Tauri 通过 `@tauri-apps/plugin-dialog` 选择文件/目录，通过 `@tauri-apps/plugin-opener` 打开结果目录，通过 `@tauri-apps/plugin-shell` 启动 `binaries/cotton-filter-backend` sidecar。
- Python 本地服务只绑定 `127.0.0.1`，前端启动时随机选择 `18763` 起的一段本地端口并等待 `/health`。
- Python 后端首次启动会在本机用户数据目录创建 `rules.sqlite3`，也可通过 `COTTON_FILTER_RULE_DB` 指定规则库路径；默认规则只初始化列名规则和颜色级值别名，不再初始化旧的评分区间和过滤区间。
- Tauri updater 已接入官方 updater plugin；发布自动更新必须使用 `TAURI_SIGNING_PRIVATE_KEY` 对安装包签名，公钥写在 `src-tauri/tauri.conf.json`。本机私钥当前位于 `/Users/zhaodesen/.tauri/cotton-filter.key`，不要提交到仓库。
- CI 构建入口是 Tauri，不要再引用已删除的 `main.py`、`cotton_filter.py`、`.github/scripts/build_windows.py` 或 Inno Setup 配置。
- CI 打包前通过 `python -m cotton_filter_app.write_build_info` 写入当前 `github.ref_name` 版本号，避免在 workflow 里写复杂跨平台内联 Python。

修改时注意：

- 保持 Tauri GUI 入口可用；不要再恢复 Tkinter 或命令行模式，除非用户明确要求。
- 保持中文日志和中文输出列名。
- 只通过文件选择入口添加 Excel 文件，不再支持拖拽导入。
- 不要让生成结果再次被输入扫描处理。
- 客户目前依赖 Excel，除非用户明确要求，不要默认改成必须部署数据库/服务端。

## 后续开发方向

优先级从高到低：

1. 扩展列名规则，适配更多客户 Excel 模板。
2. 继续丰富数据规则，如长度、强力、马值、基差、仓库、加工类型。
3. 输出更清晰的命中原因，让客户知道每个批次为什么被保留。
4. 增加预设规则，如高支纱、低杂优质、低基差、指定仓库。
5. 后续再考虑数据库、Web 筛选器、客户偏好匹配和 AI 推荐。

判断实现是否符合项目目标时，优先问：

- 是否能帮助客户更快筛出符合标准的批次？
- 是否减少了人工 Excel 判断？
- 是否保留了客户现有 Excel 工作流？
- 是否能解释每条结果为什么被选中？
